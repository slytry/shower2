Приемущества от использования

  • Упростит людям участие в проектах, позволив им изучить более структурированную историю коммитов
  • Можно создавать changelog/release notes автоматически
  • Наличие типов заставляет думать об атомарности коммитов
  • Автоматическое определение семантического повышения версии

Типы

Изменения, относящиеся к API

  1. feat - добавление нового функционала
    2. feat(radio): added default example
  2. fix - испрвление бага
    3. fix(press): skip press for contentediable element
  1. refactor/perf - реструктуризация кода,
    не меняющая никакого поведения
    2. refactor: move tabbable to if condition
  2. style - изменение в стиле кода
    3. style: update code style by prettier
  3. docs - изменения документации
    4. docs: update readme links
  4. test - изменение тестов
    5. test(datetimefield): update tests
  • build - Коммиты, которые влияют на компоненты сборки, такие как инструмент сборки, ci pipeline, зависимости, версия проекта...
    • build: update to yarn v1.22.17 (#44150)
  • ops - которые влияют на операционные компоненты, такие как инфраструктура, развертывание, резервное копирование, восстановление...
  • revert - откат на предыдущие коммиты

Что писать

Отражаем суть изменений коммита. Одно логичкески завершенное изменение - один коммит (Single Responsibility). Не получается коротко описать - скорей всего нарушен принцип едиственной ответственности

Больше глаголов. Избегаем прилагательные

Не пишем зачем, где и почему именно так

Тело

Что писать

Тело коммита может быть использовано для детализации изменений. В теле нужно сфокусируйтесь на объяснении причин изменений — на том, как всё работало до изменений и что здесь было не так, и на том, как оно работает сейчас.

Место для указания задачи

Каждая сноска должна состоять из токена, за которым следует разделитель, за которым следует строковое значение (основаное на git trailer format).
